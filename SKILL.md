# Mockup Apply

HTML 목업 → 실제 코드 적용 파이프라인. 기존 기능을 보존하면서 새 디자인/기능만 구현.

## 사용법

```
/mockup-apply <목업 HTML 경로>
/mockup-apply <목업 HTML 경로> --page <페이지명>
/mockup-apply <목업 HTML 경로> --analyze-only
```

| 인자 | 기본값 | 설명 |
|------|--------|------|
| 목업 경로 | (필수) | HTML 목업 파일 경로 |
| `--page` | (자동 감지) | 대상 페이지 파일명 (예: `backtest_detail`) |
| `--analyze-only` | false | 분석만 수행, 구현하지 않음 |

---

## Phase 1: 분석

### 1.1 목업 읽기

목업 HTML을 읽고 구조를 파악:
- 레이아웃 구조 (사이드바, 카드, 그리드 배치)
- 컴포넌트 (테이블, 차트, 버튼, 드롭다운, 탭, 모달 등)
- 스타일 (CSS 변수, 클래스명, 폰트, 색상, 간격)
- 인터랙션 (onclick, 토글, 접기/펼치기, 호버 등)
- 데이터 표시 패턴 (더미 데이터의 형식, 단위, 소수점 등)

### 1.2 현재 코드 읽기

`--page`로 지정하거나, 목업 파일명으로 자동 매핑.
매핑 불확실 시 사용자에게 확인.

현재 코드에서 파악할 것:
- 기존 컴포넌트와 레이아웃
- 데이터 로딩 로직 (DB 쿼리, API 호출)
- 이벤트 핸들러 (on_click, on_change)
- API 엔드포인트
- 번역 함수 사용 여부

### 1.3 차이점 분류

모든 요소를 세 카테고리로 분류:

```
ADD:    목업에 있고 코드에 없는 것 → 구현 대상
CHANGE: 양쪽 다 있지만 다른 것   → 수정 대상
KEEP:   코드에 있고 목업에 없는 것 → 절대 제거하지 않음
```

**KEEP 판단 기준:**
- 한/영 토글, 다크모드, 사이드바 접기 등 UI 유틸리티 → KEEP
- API 엔드포인트, 데이터 로딩 로직 → KEEP
- 이벤트 핸들러, 콜백 함수 → KEEP
- 번역 시스템 → KEEP
- 목업에 표현할 수 없는 서버사이드 기능 → KEEP

**판단 애매한 경우 → 사용자에게 확인.**

### 1.4 체크리스트 생성

분류 결과를 사용자에게 제시하고 **승인 대기**:

```markdown
## 변경 체크리스트

### ADD (새로 구현)
- [ ] 사이드바 섹션 접기/펼치기
- [ ] 태그 필터 칩 바
- [ ] 툴팁 (금융 용어 주석)

### CHANGE (수정)
- [ ] KPI 카드 레이아웃: 고정 6열 → 반응형
- [ ] 테이블 헤더 스타일 변경

### KEEP (보존, 건들지 않음)
- 한/영 토글 버튼
- 다크모드 토글
- 데이터 로딩 로직
- API 엔드포인트
```

사용자 승인 없이 구현 시작하지 않는다.
`--analyze-only`이면 여기서 종료.

## Phase 2: 디자인 시스템 동기화

### 2.1 디자인 토큰 추출

목업 HTML의 CSS에서 새 토큰 추출:
- 색상 값, 새 컴포넌트 패턴, 간격, 폰트 크기 등

### 2.2 DESIGN.md 비교 및 업데이트

프로젝트에 `docs/DESIGN.md`가 있으면 읽고 비교:
- 이미 정의된 토큰 → 그대로 사용
- 새 토큰 → DESIGN.md에 추가
- 목업 값 ≠ DESIGN.md → 어느 쪽을 따를지 사용자에게 확인

### 2.3 공통 클래스 확인

공통 CSS 파일 (layout.py 등)에서:
- 기존 공통 클래스로 처리 가능한지 확인
- 새 패턴이 2회 이상 반복되면 공통 클래스 추가

## Phase 3: 구현

### 규칙

1. **공통 클래스 우선** — 인라인 스타일로 font-size, font-weight, color 직접 지정 금지
2. **데이터 로직 보존** — DB 쿼리, API 호출, 이벤트 핸들러는 기존 코드에서 그대로 가져온다
3. **번역 유지** — 번역 함수 키를 유지/추가
4. **체크리스트 순서** — ADD 먼저, CHANGE 나중에
5. **KEEP 불가침** — KEEP 목록의 기능은 절대 제거하지 않는다

### 구현 흐름

각 체크리스트 항목에 대해:
1. 목업 HTML에서 해당 부분의 구조/스타일 참고
2. 현재 코드에서 관련 영역 찾기
3. 프레임워크 코드로 변환 (DESIGN.md CSS 변수 사용)
4. 기존 데이터 로직 연결
5. 체크리스트에서 완료 표시

**한 번에 하나씩.** 항목 하나 구현 → 확인 → 다음 항목. 한꺼번에 전부 바꾸지 않는다.

### 프레임워크별 변환 참고

프로젝트가 사용하는 프레임워크에 맞게 변환. 공통 주의사항:
- 더미 데이터를 하드코딩하지 않고 실제 데이터 소스에 연결
- 인라인 JS → 프레임워크의 이벤트 시스템으로 변환
- CSS 클래스 → 프로젝트 디자인 시스템의 공통 클래스 우선 사용

## Phase 4: 검증

### 4.1 페이지 로드 확인

서버 기동 후 해당 페이지 방문:
- HTTP 200 확인
- 스크린샷 촬영

### 4.2 콘솔 에러 체크

브라우저 콘솔에서 JS 에러/경고 수집.
새 에러 발생 시 원인 파악 후 수정.

### 4.3 KEEP 항목 검증

KEEP 목록의 기능이 모두 동작하는지 확인:
- 토글 버튼 → 클릭 후 동작 확인
- 데이터 표시 → 실제 데이터 로드 확인
- 기존 버튼/링크 → 클릭 동작 확인

### 4.4 사용자 확인

스크린샷을 보여주고 확인 요청.
문제 있으면 수정 → 재검증.

### 4.5 최종 체크리스트

```
□ 모든 ADD 항목 구현 완료
□ 모든 CHANGE 항목 수정 완료
□ 모든 KEEP 항목 동작 확인
□ 콘솔 에러 0개
□ DESIGN.md 업데이트 완료
```

---

## 주의사항

- **목업 = 디자인 가이드.** 기능 범위를 제한하지 않는다. 목업에 없는 기능도 기존에 있었다면 유지.
- **더미 데이터 ≠ 실제 데이터.** 목업의 가짜 숫자를 하드코딩하지 않고 실제 데이터 소스에 연결.
- **판단 애매하면 사용자에게 묻는다.** 특히 KEEP vs CHANGE 경계가 불확실한 경우.
